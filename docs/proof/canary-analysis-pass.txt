linkpulse api canary analysis, good release
============================================

the api rollout now runs prometheus analysis between the canary weight steps.
two AnalysisTemplates query prometheus (via the kube-prometheus-stack service)
for the canary pods only, matched by the canary replicaset pod-template hash
that the rollout injects as an arg:

  success-rate: non-5xx share of canary requests, must stay >= 0.99
  p95-latency:  p95 request latency on the canary, must stay <= 0.5s

each metric samples three times with failureLimit 2, so a healthy canary passes
and the rollout completes 20 -> 50 -> 100 on its own with no manual promotion.
the queries return real numbers from the api /metrics endpoint, which prometheus
scrapes through the ServiceMonitor shipped in the app chart.

analysisruns from a healthy release (success ratio pinned at 1.0, p95 under the
threshold):

  $ kubectl get analysisrun -n linkpulse-dev
  linkpulse-api-658459c448-5-1   Successful
  linkpulse-api-658459c448-5-4   Successful
  linkpulse-api-84588bc978-6-1   Successful

  --- analysisrun linkpulse-api-658459c448-5-1 ---
  overall phase: Successful
  metric success-rate -> Successful | measurements [1.0, 1.0, 1.0]
  metric p95-latency  -> Successful | measurements [0.475, 0.4439, 0.3307]

  --- analysisrun linkpulse-api-658459c448-5-4 ---
  overall phase: Successful
  metric success-rate -> Successful | measurements [1.0, 1.0, 1.0]
  metric p95-latency  -> Successful | measurements [0.475, 0.6795, 0.605]

the second run shows the failureLimit working as intended: two p95 samples came
in above 0.5 but under the limit of two failures, so the step still passed and
the rollout carried on.

custom alert rules loaded alongside the analysis (PrometheusRule
linkpulse-platform-alerts):

  LinkpulseApiHighErrorRate      loaded
  LinkpulseApiHighLatency        loaded
  LinkpulseWorkerStalled         loaded
  PodCrashLooping                loaded
  NodeNotReady                   loaded
